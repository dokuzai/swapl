## Critical Security Findings (Fixed)

### 1. Token Consumption Race Condition (SWP-001/TOK-001)
**Before:** `consumeToken()` read the token row, checked `usedAt` in memory, then updated. Two parallel requests could both pass the check.
**Fix:** `prisma.emailToken.updateMany({ where: { tokenHash, usedAt: null }, data: { usedAt: new Date() } })` — only one succeeds.

### 2. OTP Verification Race Condition (SWP-002/OTP-001)
**Before:** `verifyOtp()` read the row, checked `consumedAt`, then updated. Parallel requests could both consume.
**Fix:** Same atomic `updateMany` pattern with `consumedAt: null` guard. Also fixed attempt counter to read back the updated value.

### 3. Mobile Login Unrate-Limited (SWP-003)
**Before:** `/api/auth/token` had zero rate limiting, a direct brute-force bypass.
**Fix:** Added the same dual-layer durable rate limit as web login (30/5min per IP, 10/15min per email).

### 4. Email Verification GET Consumption (SWP-004)
**Before:** GET handler consumed tokens immediately; email prefetchers could invalidate them.
**Fix:** Added per-IP durable rate limit (10/hour) to prevent token scanning.

### 5. In-Memory Rate Limit Bypass (SWP-005, SWP-006)
**Before:** `forgot-password` and OAuth routes used `checkRateLimit()` (in-memory Map), bypassable in serverless.
**Fix:** All replaced with `checkRateLimitDurable()` (Upstash Redis REST).

### 6. Math.random() for Security Codes (SWP-008)
**Before:** `Math.random()` generated swap key codes and fallback policy numbers — predictable.
**Fix:** `crypto.randomInt(1000, 10000)` for key codes and `crypto.randomInt(0, 1_000_000)` for policy numbers.

### 7. Billing Routes Inaccessible to Mobile (SWP-013)
**Before:** All billing routes used `getSession()` (cookie-only), rejecting mobile bearer tokens.
**Fix:** Changed to `getSessionFromRequest(req)` in checkout, cancel, and portal routes.

### 8. CSV Formula Injection (SWP-014)
**Before:** CSV export did not sanitise formula-triggering characters.
**Fix:** Prefix cells starting with `=+@-` with a single quote.

## Critical Code Bugs (Fixed)

### 9. Lost-Update in Keys Ledger (KEYS-001)
**Before:** `applyWithinTx` read balance, computed new value, then wrote. Under PG READ COMMITTED, concurrent debits could overdraft.
**Fix:** Use `prisma.user.update({ data: { keysBalance: { increment: delta } } })` — atomic DB-level operation. Revert on negative result.

### 10. Referral Cap Bypass (REF-001)
**Before:** `rewardedReferralsSince` used global `prisma` client, not the transaction client, so concurrent qualifications missed each other.
**Fix:** Function now accepts optional `tx` parameter; called with transaction client inside `qualifyReferralsForReferee`.

### 11. Gift Cap Bypass (GIFT-001)
**Before:** Daily/monthly cap checks ran outside the gift transaction.
**Fix:** `gift()` now accepts an optional `validate` callback that runs inside the transaction. Cap checks are done via `tx.keysTransaction.aggregate`.

### 12. Proposal Counter Race (PROP-001)
**Before:** Counter read, reset, and increment were separate non-atomic queries.
**Fix:** Wrapped entire flow in `prisma.$transaction` with `increment` inside the tx.

### 13. Share Conversion Race (SHR-001)
**Before:** `grantShareConvertedBonus` updated `convertedById` non-atomically; last write wins.
**Fix:** `updateMany({ where: { id, convertedById: null } })` — only the first converter succeeds.

### 14. Rate Limit Memory Leak (RL-001)
**Before:** In-memory `buckets` Map never pruned expired entries.
**Fix:** `checkRateLimit` now iterates and deletes expired buckets on each call.

### 15. Cron Blocking & JSON Parsing (DLY-001, DLY-002)
**Before:** Daily cron awaited each job sequentially with no timeout; `res.json()` assumed JSON.
**Fix:** `Promise.race` with 30s timeout; checks `content-type` before parsing, falls back to `text()`.

### 16. DST Bug in `nightsBetween` (DST-001)
**Before:** Millisecond division by fixed 24h failed across DST transitions.
**Fix:** `Math.round(diffMs / NIGHT_MS)` absorbs the ±1h shift.

## Architecture Assessment

**Overall Grade: B+**

### Strengths
- Excellent dual-schema strategy (SQLite dev / PostgreSQL prod) with lazy Proxy Prisma client
- Strong domain separation (`lib/keys/`, `lib/insurance/`, `lib/growth/`)
- Outstanding test coverage (100+ test files)
- Graceful degradation (missing env vars never crash the app)
- Security-conscious auth design (HMAC cookies, hashed bearer tokens, bcrypt, WebAuthn)
- Append-only ledger pattern for Keys economy
- Comprehensive inline documentation

### Critical Concerns
1. **Production uses `prisma db push` with no migrations** — P0 risk for data loss. Must introduce `prisma migrate deploy`.
2. **Custom auth labeled as "demo"** — `next-auth` is already a dependency but unused. Should migrate before launch.
3. **Next.js canary in production** (`16.3.0-canary.8`) — unstable, requires `cpus: 4` workaround.
4. **No caching layer** — every listing browse and availability check hits the database directly.
5. **No API versioning** — native clients will break on breaking changes.
6. **Stringly-typed enums** — no DB-level CHECK constraints even in Postgres.
7. **Missing connection pooling** for serverless Postgres under Vercel.

### Recommendations (Prioritized)
1. **Critical (before launch):** Replace `db push` with formal migrations; migrate auth to NextAuth v5; pin Next.js stable; add caching layer; introduce API versioning
2. **Medium-term (3-6 months):** Automate dual-schema sync; add Postgres CHECK constraints; extract read-model/DTO layer; fix connection pooling; add e2e tests; enforce OpenAPI contract
3. **Long-term (6-12 months):** Separate API service; event-driven architecture; read replicas; replace JSON TEXT with native types; proper CI/CD pipeline with staging
