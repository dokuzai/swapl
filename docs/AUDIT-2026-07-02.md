# Swapl Full-App Audit — 2026-07-02

**Branch:** `audit/full-app-review`
**Method:** 12-dimension multi-agent audit (guest & host journeys pre/post approval, iOS & web design systems, Apple HIG, and five security surfaces). Every security finding was adversarially verified against the real code before being kept; the two criticals were additionally re-read by hand. Findings de-duplicated against the prior 2026-06-19 audit (`MASTER-REPORT.md`) — already-fixed items are excluded; still-open items are carried forward and marked.
**Codebase at audit:** 157 API routes, 61 Prisma models, ~60 SwiftUI feature views.

---

## Executive summary

The platform is in good shape on the fundamentals. The **reciprocal home-swap** flow is complete and polished on both web and iOS (proposal → agreement → shared trip cockpit → check-in/out → review → dispute). Security is genuinely strong: the six hardening fixes from the June audit are all still in place, authentication crypto and OAuth verification are correct, authorization/IDOR is clean across ~35 sampled routes (all 13 admin routes enforce `role==='swapl_admin'`), and the privacy invariants (coordinate fuzzing, contact-channel reveal gating, DOB confidentiality) hold.

Two themes dominate the findings:

1. **Two confirmed critical money bugs in the Keys economy.** The `KeysStay` confirm/cancel/decline state machine runs at READ COMMITTED with a non-locking status re-read (unlike `createKeysStay`, which uses Serializable). Concurrent requests can double-run the ledger, **minting Keys from nothing** — both a guest self-mint (double-release) and a host self-mint (double-confirm). These corrupt the points economy and must be fixed before launch.

2. **The Keys (points) stay is a second-class citizen end-to-end.** Independently surfaced by the guest, host, and security agents: a confirmed Keys stay never transitions to `completed`, so it never appears in trip history; the guest never receives the home guide; there is no check-in/out, no dispute path, and no review for either side — despite a cover policy being issued. The reciprocal swap has all of this; the paid points stay has almost none of it.

Beyond those, the design system is "aspirational" on both platforms — the token pipeline exists but is largely bypassed (web never imports the token package; iOS's new appearance-toggle is bypassed app-wide so the shipped "Apple look" feature is effectively broken), and the iOS app has two real Apple-HIG gaps (Dynamic Type is completely non-functional; no Dark Mode).

## P0 — Critical (fix before launch)

### SEC-KEYS-01 — Double-release of a pending Keys stay mints free Keys (TOCTOU) · CRITICAL · confirmed
`releaseKeysStay` (`app/lib/keys/stay.ts:366`) opens `prisma.$transaction` with **no isolation level** (READ COMMITTED on the prod Postgres adapter, `app/lib/db/prisma.ts:9-19`), re-reads status with a non-locking `findUnique` (`stay.ts:368`), then credits `release(guestId, keysCost)` (`stay.ts:374`). There is no unique constraint on `KeysTransaction(stayId, kind)` and ledger rows carry `eventKey=null`, so a duplicate release row is permitted. `createKeysStay` by contrast uses `{ isolationLevel: Serializable }` (`stay.ts:235`). The cancel (`.../cancel/route.ts:16`) and decline (`.../decline/route.ts:16`) routes are unthrottled.
**Impact:** A guest cancel racing a host decline (or two concurrent cancels) both release the single hold → guest ends up `+keysCost` richer than before booking. Repeatable per booking; minted Keys are spendable on real stays where a real host earns. Triggerable without controlling both requests.
**Fix:** Gate the flip with a conditional `updateMany({ where: { id, status: 'pending' }, data: { status: outcome } })` and run the ledger release only when `count === 1`, inside the transaction — plus a per-`(stayId, transition)` idempotency `eventKey` on the ledger row. (Serializable alone also closes it, matching `createKeysStay`.)

### SEC-KEYS-02 — Double-confirm of a pending Keys stay double-credits the host (TOCTOU) · CRITICAL · confirmed
`confirmKeysStay` (`app/lib/keys/stay.ts:311`) has the same shape: default isolation, non-locking `findUnique` (`stay.ts:314`), then `release + spend + earn` (`stay.ts:323-325`) and a status update `where:{ id }` that does **not** gate on current status (`stay.ts:328`). Confirm route (`.../confirm/route.ts:19`) is host-auth'd but unthrottled.
**Impact:** Two concurrent confirms both execute the ledger triple. Net per confirm: guest 0, host `+keysCost`. A second racing confirm credits the host an extra `keysCost` with no matching guest debit — the host mints Keys.
**Fix:** Same as SEC-KEYS-01: conditional `updateMany` status gate + idempotency `eventKey` (or Serializable), all in one transaction.

> These two share one root cause and one fix pattern; treat as a single work item touching `confirmKeysStay` + `releaseKeysStay` + a `KeysTransaction` unique constraint / eventKey scheme.

## Cross-cutting theme — the Keys (points) stay is second-class

The reciprocal swap has a full lifecycle; the paid Keys stay is missing most of it. This is the single highest-leverage product gap and spans several findings:

- **JRN-GP-02 / JRN-HP-01 — never reaches `completed`.** No route or cron ever writes `KeysStay.status = "completed"` (only `pending→confirmed|declined|cancelled` and a date-change edge in `lib/date-change.ts:347`). `lib/story.ts:74-90` *reads* completed stays, so a host's hosting history and any completion-gated UI are permanently empty. There is an `agreements-complete` cron for swaps but nothing for `KeysStay`.
- **JRN-GP-01 — no review path.** `SwapReview` is keyed only to `SwapAgreement` (`schema.prisma:824-847`); the only review route requires `agreement.status==='COMPLETED'`. A points guest/host can never rate each other → zero reputation accrual for the points economy.
- **JRN-GP-03 / JRN-HP-05 — no dispute path.** Disputes are `SwapAgreement`-keyed only (`agreements/[id]/dispute/route.ts:31-39`); `KeysStay` has no dispute route and `KeysStayDetailView` no affordance — despite a cover policy being issued on confirm (`stay.ts:271-306`) and advertised on the banner.
- **JRN-GP-04 / JRN-HP-04 — no home guide for the guest.** `keys/stays/[id]/route.ts:26` loads `homeGuide` but the response never returns it; the guest arrives with only an address.
- **JRN-HP-03 — request notifies host by push only** (`keys/stays/route.ts:101`), no email fallback; held Keys can sit `pending` indefinitely (no expiry).
- **JRN-HP-02 — hosting missing from "Ways to earn Keys"** (`lib/keys/earn-ways.ts`) though `earn_host` is the main repeatable earn.
- **JRN-HP-07 — web parity:** rich Keys-stay detail (address/contacts/map/chat) exists only on iOS; web is a bare confirm/decline card.

**Recommended epic:** "Keys-stay lifecycle parity" — completion cron → unlock review + story + reconciliation; return home guide; generalize disputes to accept a `KeysStay`; add email notification + hold expiry; add the earn entry; build the web detail view.

## Design system

### iOS
- **DS-IOS-01 · HIGH — the new `SwaplTheme` appearance toggle is bypassed app-wide, so the shipped "Apple look" is effectively broken.** Feature code reads `@Environment(\.swaplTheme)` in only 4 files but references `SwaplSemanticLight.*` 554× and `AirbnbPalette.*` 864×; even the appearance picker's own rows are hardcoded to the brand palette (`AccountSettingsViews.swift:111-131`). Picking "Apple" changes only the scroll canvas — cards/chat/chips/sheets stay cream/navy. The settings copy promises it "re-skins every screen." Fix: route content colors through `theme.*` (prioritize the picker + high-traffic surfaces), or hide the toggle until content colors are tokenized.
- **DS-IOS-02 · MED — three overlapping token systems** (`SwaplColor`, `SwaplSemanticLight`, `AirbnbPalette`; radii `SwaplDesignSystem.CornerRadius` vs generated `SwaplRadius` with different numbers). Fix: pick one scale; make `SwaplDesignSystem` re-export generated tokens; demote `AirbnbPalette` to a thin shim.
- **DS-IOS-03 · MED — inconsistent card corner radii** (Keys wallet stacks 32/28/20 in one scroll; chat bubbles hardcode 18/22/28). Fix: standardize on one card token / `SurfaceCard`.
- **DS-IOS-04 · MED — shared components barely adopted** (`SurfaceCard` in 3 files; the `.background(...).overlay(...)` card pattern re-hand-rolled dozens of times without the shadow). Fix: adopt `SurfaceCard`/`PrimaryPill` as default.
- **DS-IOS-05 · MED — screen background & header conventions diverge** (`.swaplScreenBackground()` in 31/58 files; headers split across `SwaplPageTitle`/floating-glass/system nav with no rule for peers). Fix: document and enforce the rule.
- **DS-IOS-06 · LOW — raw `Color.white` surface fills** in several views bypass tokens. Fix: use `theme.card`.

### Web
- **DS-WEB-01 · HIGH — the web app never consumes the design-token package.** `@swapl/design-tokens` is not a dep of `app/` and imported nowhere; `globals.css:14-24` hand-declares the same palette. Fix: emit CSS vars from tokens + `@import`, add the dep, or at minimum a drift test.
- **DS-WEB-02 · HIGH — palette triplicated by hand** across `core.json`, `app/globals.css`, `marketing/globals.css` — silent drift. Fix: shared emitted CSS artifact imported by both apps.
- **DS-WEB-03 · HIGH — shadcn Button/Card are effectively dead** (Button 1×, Card 5×) vs `.surface-card` 189× / `.pill-*` 142×; two parallel button systems, the CSS one lacking disabled/focus/dark states. Fix: pick one, fold + migrate.
- **DS-WEB-04 · HIGH — 57 hardcoded `#dc2626` error colors** bypass the `--destructive` token and don't flip in dark mode (login-form, auth-providers, inspire-client, travel-windows editor). Fix: `text-destructive` utility or a `<FormError>` component.
- **DS-WEB-05 · MED — no typographic scale:** ~399 arbitrary `text-[NNpx]` sizes + ad-hoc `tracking-[...]`. Fix: define size/tracking scale in tokens, codemod, lint.
- **DS-WEB-06 · MED — inline `style={{…var(--token)}}` dominant (~963×); `@theme` utilities unused.** Fix: migrate to `className` utilities + ESLint rule.
- **DS-WEB-07 · MED — generated token outputs hand-edited/drifted** (CityPalette type, Swift `Hashable/Sendable`, invalid `2xl` id; `font.weight` missing in Swift/Kotlin; radius `3xl/4xl` only in CSS); no CI drift gate. Fix: make `build.mjs` emit what's checked in + add `check:drift` CI.
- **DS-WEB-08 · LOW — arbitrary radius drift** (`rounded-[22px]` in nav/footer/admin). Fix: token or snap to scale.

## Apple HIG (iOS)
- **HIG-01 · HIGH — Dynamic Type is completely non-functional.** `SwaplFonts.swift:28-45` defines fonts via `.custom(family, size:)` with no `relativeTo:`; 765 helper call-sites + 215 `.font(.system(size:))`, zero scalable styles. Raising system/accessibility text size changes nothing — a common App Store accessibility rejection and a lock-out for low-vision users. Fix: add `relativeTo:` text-style mapping; convert `.system(size:)` sites; add `minimumScaleFactor` where single-line clips; test at AX5.
- **HIG-02 · MED — no Dark Mode; app hard-forces Light** (`SwaplApp.swift:113 .preferredColorScheme(.light)`; both appearances resolve Light; Dark tokens are dead code). Fix: either commit to Light-only and delete dead Dark tokens, or route content colors through `SwaplTheme` and drop the forced light. (Couples with DS-IOS-01.)
- **HIG-03 · MED — primary CTA label contrast fails WCAG AA.** White on brand pink `#F24B8E` at 15pt-medium = 3.4:1 (AA needs 4.5:1; large-text 3:1 needs ≥18pt/≥14pt-bold). Repeated on every primary pill, empty-state CTA, selected chip. Fix: darken pink toward ~`#D81B60` or make labels ≥14pt bold; rebuild tokens.
- **HIG-04 · LOW — near-total absence of haptics** (one call in the app). Fix: `.sensoryFeedback` on publish/accept/OTP/errors/destructive confirms.
- **HIG-05 · LOW — custom floating-glass header replaces the system nav bar on 12 pushed screens**, forfeiting large-title collapse, long-press-back, VoiceOver back semantics; edge-swipe kept via a fragile UIKit gesture hack. Fix: prefer a styled system nav bar; reserve custom header for full-bleed heroes; add `.isButton` trait where it stays.
- **HIG-06 · LOW — Sign in with Apple parity depends on a server flag.** `LoginView.swift:130-145` shows Apple only `if providers.apple`; a `{google:true, apple:false}` response ships a Guideline 4.8 violation. Fix: make SIWA unconditional or gate Google on `providers.apple`; add a test.

## Security

**Strong overall.** Authz/IDOR is clean across ~35 sampled routes; all 13 admin routes enforce `role==='swapl_admin'`; cron + both webhooks fail closed with signature/secret checks; input validation, SSRF defenses, uploads, and AI cost controls are solid; SWP-015 (register Zod bypass) is now **fixed**. Findings below are the Keys criticals (above) plus lower-severity items and carried-forward known-open issues.

- **SEC-KEYS-03 · MED (verified; downgraded from high) — gift daily/monthly caps bypassable under concurrency.** `keys/gift/route.ts:72-91` checks caps via `aggregate` inside a READ COMMITTED tx, which sees only committed rows; concurrent gifts both pass. Per-transfer cap (Zod, 50) still bounds each call. Fix: Serializable or an atomic rolling-window counter; fix the misleading comment.
- **SEC-KEYS-04 · LOW — welcome-bonus grant not concurrency-safe on its own** (`ledger.ts:269-287`, `eventKey=null`); mitigated upstream by the verification status machine. Fix: deterministic `eventKey = welcome_bonus:<userId>` for the unique constraint.
- **SEC-AUTH-01 · MED — OTP-verify + passkey-login use the non-durable in-memory rate limiter** (`otp/verify/route.ts:12,24`; passkey login options/verify), a no-op across serverless. Per-code 5-attempt cap still holds; the per-IP throttle does not. Fix: switch to `checkRateLimitDurable`.
- **SEC-AUTH-02 · MED (SWP-020, known-open) — web session is stateless/non-revocable up to 30 days**, surviving password change and suspension (`session.ts:55-80`; no `passwordChangedAt`/`sessionEpoch`). Fix: add `sessionEpoch` to `User`, stamp into cookie, reject stale; bump on password change + suspend.
- **SEC-PRIV-01 · MED (SWP-007, known-open) — plaintext at-rest secrets** `aiApiKey`/`keyCode1/2`/`wifiPassword` (`schema.prisma:46,708-709,751`). Access-gated correctly; storage is not. Fix: app-layer `aes-256-gcm` envelope + KMS.
- **SEC-PRIV-02 · LOW (new) — pending co-traveler email disclosed to all thread participants.** `proposals/[id]/participants/route.ts:87-98` returns raw `invitedEmail` to any `canAccessConversation` viewer (incl. active guests). Fix: return `invitedEmail` only to principals, or mask.
- **SEC-INPUT-01 · LOW — unguarded `JSON.parse` on `photos`/`destinations` columns** (`agreements/[id]/trip/route.ts:103`; `trip/check-event.ts:69,87,132`; `travel-windows/route.ts:40`) deviates from the `parseJSON()` convention; latent 500 risk, not attacker-triggerable. Fix: route through `parseJSON<string[]>(v, [])`.
- **SEC-AUTH-03…07 · LOW (known-open)** — SWP-010 `__Host-` cookie prefix; SWP-012 passkey `requireUserVerification:false`; SWP-016 password min-6 / bcrypt cost 10; SWP-015-adjacent register reads `platform`/`appVersion` from raw body; SWP-009 register 409 enumeration. See `MASTER-REPORT.md` remaining-work list.
- **SEC-AUTHZ notes (low, non-exploitable):** admin routes split between cookie-only `requireAdmin()` and bearer-capable `requireAdminFromRequest()` (fails closed for mobile admin); `proposals/[id]/participants` POST lets a principal add any userId to their own thread with no invitee consent step.

**Verified clean (no action):** coordinate fuzzing on every listing path; contact-channel reveal gating; DOB self-only; OAuth signature/iss/aud/exp verification; atomic token/OTP consume; bearer-token hashing + revoke; SSRF host allowlist + `redirect:"error"`; SVG upload exclusion; AI output re-validation; no raw SQL; Stripe/Didit webhook signatures + idempotency.

## Appendix — test-harness regression (found during branch merge)
37 tests across 11 files fail on `main` (e.g. `keys-ledger`, `plan-limits`, `growth-referrals`, `inspire`, `swap-reviews`) — the mock Prisma harness returns `{increment:N}` objects where tests expect scalars, and some route tests hit `prisma.swapAgreement.findMany` on an undefined mock. **These fail identically at the pre-merge commit `fdc0e5f`**, so they are not a merge regression, but they mean the suite is not currently a reliable gate. Recommend a P1 "repair test harness" task before the dev wave leans on it.
